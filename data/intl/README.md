# `data/intl/` — vendored international-market series

Weekly series backing the six international analyst personas (`ea_rates`,
`uk_rates`, `jp_rates`, `ea_equity`, `uk_equity`, `jp_equity`). Loaded by
`src/data/intl_local.py` and referenced by `personas/<driver>.yaml` through the
closed feature-op vocabulary.

## Provenance

Generated by `scripts/build_intl_series.py` from the sister repo's Bloomberg
terminal export `../berkeley-mfe-blackrock-2026/total_assets_weekly.csv`
(weekly Friday closes, committed there via upload — no fetch script exists
upstream). One CSV per series, two columns `date,<SERIES_ID>`, 1,327 Fridays
2001-01-05 → 2026-06-05 (the `*_RVOL13` series start 13 weeks later).
Bloomberg data is licensed; these are derived weekly extracts vendored for an
academic class project — do not redistribute beyond the team.

## Point-in-time contract (important)

Each series is a **market Friday close**, observable on that same Friday — the
observation date IS the decision date. So `intl_local.load_series` applies
**no** `PUBLICATION_LAG_DAYS` shift (same contract as `data/equity/`, unlike
`fred_local`). Do not add an entry for any `INTL_*` id to the FRED lag table.
Tokyo and London Friday closes precede a New York decision point, so same-day
observability holds strictly.

## Why vendored features, not raw inputs

The op vocabulary (`src/layered/features/ops.py`) deliberately has no
rolling-std, so annualized realized vol cannot be expressed in a persona spec.
The three `*_RVOL13` series (13-week rolling std of weekly percent returns,
× √52) are computed in `build_intl_series.py` and read with `op: level`.

## Series

| series id | Bloomberg source | what it is | driver(s) |
|---|---|---|---|
| INTL_DE2Y | GTDEM2Y Govt | 2y German Schatz yield, % | ea_rates |
| INTL_DE10Y | GTDEM10Y Govt | 10y German Bund yield, % | ea_rates (level/clock) |
| INTL_UK2Y | GUKG2 Index | 2y UK gilt yield, % | uk_rates |
| INTL_UK10Y | GUKG10 Index | 10y UK gilt yield, % | uk_rates (level/clock) |
| INTL_JP2Y | GJGB2 Index | 2y JGB yield, % | jp_rates |
| INTL_JP10Y | GJGB10 Index | 10y JGB yield, % | jp_rates (level/clock) |
| INTL_US10Y | USGG10YR Index | 10y US Treasury yield, % | all rates (cross-market spread) |
| INTL_EURUSD | EURUSD Curncy | USD per euro | ea_rates, ea_equity |
| INTL_GBPUSD | GBPUSD Curncy | USD per pound | uk_rates, uk_equity |
| INTL_JPYUSD | JPYUSD Curncy | USD per yen | jp_rates, jp_equity |
| INTL_SXXGV | SXXGV Index | STOXX Europe 600 gross TR, EUR | ea_equity (level/clock) |
| INTL_UKX | UKX Index | FTSE 100 price index, GBP | uk_equity (level/clock) |
| INTL_MSCIJP | GDDUJN Index | MSCI Japan gross TR, **USD** | jp_equity (level/clock) |
| INTL_SXXGV_RVOL13 | derived | 13-wk realized vol, annualized | ea_equity |
| INTL_UKX_RVOL13 | derived | 13-wk realized vol, annualized | uk_equity |
| INTL_MSCIJP_RVOL13 | derived | 13-wk realized vol, annualized | jp_equity |

Index-construction caveats the personas state honestly: `INTL_UKX` is a
**price** index (dividends excluded); `INTL_MSCIJP` is **USD-denominated**, so
its moves mix Japanese equities and the yen (`INTL_JPYUSD` is a declared
feature of `jp_equity` for exactly this reason). There are no replay CSVs for
international drivers — they produce views only via a live LLM feed, which is
a new experiment (prereg + cost approval) whenever it runs.
